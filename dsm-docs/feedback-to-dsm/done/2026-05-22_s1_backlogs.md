## BL proposal: Adopt "smoke tests" as a DSM artifact, with a canonical location and per-item logging convention

### Problem

DSM_0.2 and DSM_4.0 prescribe several mechanisms for verifying code behavior:

- Unit tests (`tests/unit/`, `pytest tests/`) — run at sprint boundaries or in CI
- `/verify` skill — runs the assembled app and observes behavior at a milestone
- Sprint Boundary Checklist (DSM_2.0.C Template 8) — "Tests passing"

None of these cover **per-item, in-the-moment smoke testing as each file is built**, where the smoke-test commands and their expected outputs are recorded to a persistent location in the repo. This is a recognized industry practice with established terminology and a clear shape; DSM simply does not name a slot for it today.

### Why this matters — observed in S1

S1 of `traveline-ds-project-skeleton` (2026-05-22, a Python ML skeleton being built ~4 hours before a live pair-coding interview). After committing Item 6 (`loader.py`) and Item 7 (`transform.py`), the user asked **"how can I validate manually both loader.py and transform.py?"**. The agent produced ~80 lines of validation commands inline. The user ran them, all passed, then asked where to persist them — there was no canonical DSM location.

The validation commands existed in the agent's transcript-context but nowhere on disk that survived the session. The same pattern will recur at every item in this build (8 more items to go) and at every similar build in any other DSM spoke. The failure mode is silent: the agent re-derives the same commands at every session boundary, the user has no recovery procedure if a component breaks weeks later, and the git log loses its "this was working at each step" property because the evidence lives in volatile transcript context.

### "Smoke test" is the established industry term for this

The practice this BL proposes adopting is not a new invention. The single most widely recognized term is **smoke test**: a small, fast check run after each change to confirm the basic thing still works. Origin: hardware engineering ("plug the new circuit in, see if it smokes"). Popularized for software by Microsoft's daily-build-and-smoke-test practice in the 1990s. Documented in Steve McConnell's *Code Complete* Ch. 22. Standard term in modern CI/CD ("smoke tests pass before deploy").

Adjacent industry practices in the same family (overlapping but distinct):
- **Runbooks** (Google SRE Book) — persistent, repo-versioned procedures for verifying or operating a system; same structural form, broader scope than build-time smoke tests
- **Engineering logbooks / lab notebooks** (Karpathy's *Recipe for Training Neural Networks*) — ML-research-specific variant focused on experiment results
- **REPL-driven validation** (scientific Python, Clojure) — the same per-item-check practice, usually not persisted
- **Acceptance test logs** (IEEE 829-2008, ISO 9001, IEC 62304) — heavyweight formalization for regulated industries

"Smoke test" is the right name because it is the most general, widely understood, and language-agnostic of these. This proposal aligns DSM with established industry practice rather than coining a new term.

The `guides/` folder is already canonical (`/dsm-go` Step 0.5 lists it among the 9 canonical `dsm-docs/` subdirs). Only the filename needs blessing. Lazy creation — file exists only when there are smoke tests to record. Append-only; one section per file/component validated.

### Distinction from `/verify`

`/verify` (per the verify skill's description) is for running the assembled app and observing behavior at a milestone — high-level integration check, executed by the agent. Smoke tests are per-file, per-item, written down as commands the user can run themselves. They are complementary; this proposal does not replace, alter, or interfere with `/verify`.

### Distinction from unit tests

Unit tests run in CI and at sprint boundaries; they exercise functions and methods through pytest. Smoke tests exercise the user-facing surface (CLI args, install commands, error messages, integration paths) and are run by a human from a shell. The two are complementary: a smoke test that involves five sequenced shell commands and an inspected output is not pytest-shaped; a pytest unit test asserting `transformer.fit(X).transform(Y).shape == (n, 30)` is not smoke-test-shaped.

The argument for keeping smoke tests *in addition to* unit tests, not instead of:
1. **In scaffolding builds, test infrastructure is itself being built.** Items 1-8 of a 16-item skeleton may land before `tests/` exists. Smoke tests cover those items immediately; pytest catches up once the test runner exists.
2. **Smoke tests catch user-environment integration breakage.** A pytest can stub the venv path; the user running the command from their actual `.venv/` can't. The discrepancy is sometimes the point (e.g., catching that `python3.11` works but the system `python3` is 3.10).
3. **Smoke tests are reusable as onboarding docs and recovery procedures.** A new contributor (or future-you) can `grep` the smoke-tests log to find "how do I verify the loader works?" without spinning up the test suite.
4. **The git log + smoke-tests log together form a verifiable build narrative.** "Each commit was preceded by a documented smoke-test run" is a stronger demo/portfolio claim than "the test suite passes today".

### Priority

Medium. The methodology works without this codification — but the failure mode it closes (smoke-test commands living only in agent transcript-context, dying with the session) recurs in any project that has manual setup steps, and the artifact is cheap to specify. Naming the practice now also pre-empts five different projects coining five different names for the same thing.

### What this proposal does NOT ask for

- Not a replacement for unit tests
- Not a replacement for `/verify`
- Not a hard per-item gate (sprints without new components don't need new smoke tests)
- Not a new skill (could be added later as a companion if useful — e.g., `/dsm-smoke-test <item>` to append a templated section)
- Not a change to `/dsm-go` or `/dsm-wrap-up` (smoke tests are author-driven, not session-lifecycle-driven)
